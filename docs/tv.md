## What Is the TV App?

A **Next.js 15 web app** running inside an Android TV WebView APK (API 26+). It targets cinema lobbies and hotel displays. The WebView wraps the web app and handles:

- Hardware key events (D-pad, remote)
- Autoplay permissions
- Encrypted storage for device tokens
- OTA APK updates

## Dev Setup

```bash
# Start TV app (must bind to 0.0.0.0 for Android emulator access)
pnpm dev:tv
# → Next.js on port 3004, listening on 0.0.0.0

# From Android emulator or real device on same LAN:
http://10.0.2.2:3004      # emulator only (10.0.2.2 = host machine)
http://<your-LAN-IP>:3004 # real device on same WiFi
```

**Always ensure the API is running too:** `pnpm dev:api` (port 3001)

## Device State Machine

```
BOOTING
  ↓
INITIALIZING  (load encrypted prefs, check device token)
  ↓              ↓
UNPAIRED    PAIRED
  |             ↓
  |          SYNCING   (fetch schedule, TvConfig, TvMacro)
  |             ↓
  |          ACTIVE    ← normal operation
  |             ↓ ↑
  |          OFFLINE   (no network — show offline screen)
  |          UPDATING  (OTA APK update in progress)
  |          ERROR     (unrecoverable — show error screen)
  |
  └──── (QR scan / PIN entry) ──→ PAIRED
```

### Key Components per State

| State | Component |
|-------|-----------|
| UNPAIRED | `screens/pairing-screen.tsx` |
| OFFLINE | `screens/offline-screen.tsx` |
| SYNCING | `screens/syncing-screen.tsx` |
| ACTIVE | `layout/tv-shell.tsx` |
| ERROR | `screens/error-screen.tsx` |

## Pairing Flow (Step by Step)

### 1. TV Side (UNPAIRED state)
- TV shows QR code + 6-digit PIN
- QR encodes: `https://app.neofilm.io/pair?code=<pairing-code>`
- Pairing code generated by `POST /auth/device/generate-pairing-code`
- TV polls `GET /auth/device/pairing-status?code=<code>` every 5s

### 2. Partner Side
- Partner opens `/partner/screens/[screenId]/pairing`
- Scans QR or enters PIN → `POST /auth/device/pair`
  ```json
  { "screenId": "...", "pairingCode": "ABC123" }
  ```
- Backend: creates `Device`, issues device JWT, links to `Screen`

### 3. Completion
- TV polling detects status `paired` → receives device JWT
- TV stores JWT encrypted in Android SharedPreferences
- TV connects to WebSocket `/devices` with JWT
- TV state → PAIRED → SYNCING → ACTIVE
- Partner dashboard shows screen ONLINE in real-time

## Ad Playback System

### Ad Queue Hook (`use-ad-queue.ts`)

1. Fetches `GET /tv/ads?screenId=<id>` → active `Campaign[]` targeting this screen
2. Builds ad rotation queue from `CampaignTargeting.includedScreens`
3. Plays each ad for `TvMacro.spotDurationMs` (default: 30000ms)
4. After each play → writes `DiffusionLog` to `POST /tv/diffusion`
5. Rotate to next ad (round-robin)

### Real-Time Ad Updates

```
Campaign published
  → POST /campaigns/:id/publish
  → DeviceGateway.pushToScreen(screenId, 'tv:ads:update')
  → TV receives 'tv:ads:update'
  → TV calls GET /tv/ads
  → Queue updates immediately
```

### Ad Interstitial (`common/ad-interstitial.tsx`)

Full-screen overlay that shows during ad spot. Features:
- Skip button (after `TvMacro.skipDelayMs` if > 0)
- Countdown timer (`common/countdown-timer.tsx`)
- Autoplay `<video muted autoplay>` — both attributes required

## IPTV Playback (`common/iptv-player.tsx`)

Uses **hls.js** for HLS stream playback:

```typescript
import Hls from 'hls.js';

if (Hls.isSupported()) {
  const hls = new Hls();
  hls.loadSource(channel.streamUrl);
  hls.attachMedia(videoElement);
} else {
  // Fallback: native HLS (Safari)
  videoElement.src = channel.streamUrl;
}
```

**M3U8 playlist source:** `IPTV_M3U_URL` env var → parsed into `TvChannel[]` records.

## WebSocket Connection (TV Side)

```typescript
import { io } from 'socket.io-client';

const socket = io(process.env.NEXT_PUBLIC_WS_URL, {
  path: '/socket.io',
  query: { deviceId, token: deviceJwt },
  transports: ['websocket'],   // No polling — saves bandwidth
  reconnectionDelayMax: 300000, // Max 5-minute backoff
});

socket.on('tv:ads:update', () => refetchAds());
socket.on('schedule', (slots) => updateSchedule(slots));
socket.on('command', ({ type }) => handleCommand(type));
socket.on('tv:macros:update', (macro) => updateMacro(macro));
```

**MQTT Fallback:** If WebSocket is unavailable, TV connects to MQTT broker (`MQTT_BROKER_URL`) on topic `device/<deviceId>/commands`.

## Heartbeat (TV → Server)

Every 30 seconds:
```typescript
socket.emit('heartbeat', {
  deviceId,
  isOnline: true,
  appVersion: process.env.NEXT_PUBLIC_APP_VERSION,
  uptime: performance.now() / 1000,
  cpu: getCpuUsage(),     // TODO: verify availability in WebView
  memory: getMemUsage(),
  temperature: null,       // Android TV specific
});
```

Server updates `DeviceHeartbeat` + `ScreenLiveStatus.isOnline = true`.

If heartbeat not received for 90s → `ScreenLiveStatus.isOnline = false` → PartnerGateway notifies partner.

## Android WebView Constraints

| Constraint | Impact | Workaround |
|-----------|--------|-----------|
| **Autoplay policy** | Video won't play without user gesture | Always add `muted autoplay` attributes; WebView grants autoplay to trusted origins |
| **Mixed content** | HTTP resources blocked on HTTPS page | In prod: serve all assets over HTTPS; in dev: both HTTP |
| **D-pad navigation** | No mouse/pointer events | Use `onKeyDown` handlers for navigation; avoid hover-only states |
| **User Agent** | Contains `"Android TV"` | Use for layout adaptation: `navigator.userAgent.includes('Android TV')` |
| **localStorage** | Available but not encrypted | Sensitive tokens (device JWT) must be in native Android EncryptedSharedPreferences |
| **No service workers** | Cache must be handled by native | APK handles disk cache for video assets |

## OTA Updates

1. Server pushes `command: { type: 'update', apkUrl, sha256 }`
2. TV app notifies native WebView bridge
3. Native layer: download APK, verify SHA-256
4. If hash mismatch → reject, notify server
5. If valid → install, reboot
6. On failure → rollback to previous version, emit error event

## Debug Checklist

### TV App Shows Blank Screen

```
□ Is TV app running?
  curl http://10.0.2.2:3004

□ Is API running?
  curl http://10.0.2.2:3001/api/v1/health

□ Check browser console (Chrome DevTools → chrome://inspect → inspect WebView)

□ Is device token valid?
  Check DevicePairingRequest table — token expired?

□ Any JS errors in console?
```

### TV Shows Pairing Screen (but already paired)

```
□ Encrypted prefs cleared? (app reinstall, factory reset)
□ Device token expired? (JWT expiry)
□ Screen deleted from partner portal?
□ Re-pair: partner opens /partner/screens/[id]/pairing
```

### No Ads Showing

```
□ Campaign status = ACTIVE?
  SELECT status FROM "Campaign" WHERE id = '...';

□ Screen in CampaignTargeting.includedScreens?
  Check CampaignTargeting relation

□ Device connected to /devices namespace?
  DeviceGateway.isDeviceConnected(deviceId)

□ TvMacro configured?
  GET /tv-config/:screenId

□ Ad assets accessible from device?
  curl <CDN_BASE_URL>/<s3Key>
```

### IPTV Stream Not Loading

```
□ hls.Hls.isSupported() → false? Try native src fallback
□ M3U8 URL reachable from device network?
  curl <IPTV_M3U_URL>
□ CORS headers on stream server?
□ Mixed content? (HTTPS page loading HTTP stream)
```

### Real-Time Not Working

```
□ WebSocket connected? Check Network tab → WS frames
□ Device token valid for /devices namespace?
□ pushToScreen() called with correct screenId (not deviceId)?
□ Event name matches: 'tv:ads:update' (case-sensitive)
□ Redis adapter running? (production only)
```

### Android Emulator Network Issues

```
□ Using http://10.0.2.2:3004 (not localhost)?
□ TV app bound to 0.0.0.0? (dev script: --hostname 0.0.0.0)
□ Try cold boot of emulator
□ Check Android network security config allows cleartext (dev only)
□ Emulator API level ≥ 26?
```
